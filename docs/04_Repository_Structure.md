## Workspace package conventions

Every workspace package has:

- `package.json` with `"name": "@caeorta/<name>"`
- `tsconfig.json` extending `tsconfig.base.json`
- A `src/` directory for source
- A `README.md` explaining what the package is for and its primary exports

Apps reference packages via workspace protocol:
```json
"dependencies": {
  "@caeorta/types": "workspace:*",
  "@caeorta/supabase": "workspace:*"
}
```

## Branch strategy

- `main` is always deployable. Web deploys automatically from main; mobile uses EAS Update for JS-only changes.
- Feature branches: `feat/short-description`, `fix/short-description`, `chore/short-description`
- PRs in the App area are reviewed by Sulaiman before merge. PRs in the Platform area are reviewed by Sulaiman before merge. Both founders must agree on cross-cutting changes (schema, agent contract, CI architecture).
- Squash merge into main
- No direct commits to main except in genuine emergencies; document if it happens

Enforce squash-only merge in GitHub repo settings:

1. github.com/Caeorta-io/caeorta_app/settings
2. Under "Pull Requests": disable "Allow merge commits" and "Allow rebase merging".
3. Leave only "Allow squash merging" enabled.
4. (Optional, recommended) check "Automatically delete head branches" so merged feature branches get cleaned up automatically.

Branch protection itself is unavailable on the Free plan for private repos; squash-only merge is enforced at the repo-settings level (cosmetic, not preventive) and CI runs visibly on PRs.

## Commit conventions

[Conventional Commits](https://www.conventionalcommits.org/):

- `feat:` new feature
- `fix:` bug fix
- `chore:` tooling, dependencies, cleanup
- `docs:` documentation only
- `refactor:` code restructuring without behavior change
- `test:` test additions or changes
- `perf:` performance improvement

Examples:
- `feat(mobile): add QR scanner to pairing flow`
- `fix(admin): correct device offline timestamp display`
- `chore: bump expo to SDK 53.0.5`

## Environments

Three environments. Strictly separated.

### Local development
- Muhammed runs Expo Go on his phone (or simulator/emulator) hitting his local machine
- Local machine connects to **dev Supabase**
- Edge Functions tested locally via `supabase functions serve`
- Migrations applied to dev Supabase via `supabase db push`

### Preview
- EAS preview builds (TestFlight internal-internal use, ad-hoc Android APKs)
- Hits **dev Supabase**
- Used for designer reviews, integration tests with AI agent project, internal demos

### Production
- Built only from `main` branch
- Hits **prod Supabase**
- Distributed via TestFlight (iOS) and Play Internal Testing (Android) to pilot users
- Sentry release tracking enabled
- PostHog production project

## Secret management

- All production secrets stored in 1Password (or equivalent), shared vault between two founders
- API keys rotated quarterly; date of last rotation tracked in 1Password
- Apple signing certificates and Google signing keys stored in 1Password
- **Critical: losing Google's app signing key means losing the ability to update the Play Store listing forever.** Treat this with extreme care.

## CI/CD pipeline

GitHub Actions handles:

1. **On every PR:**
   - Lint (ESLint)
   - Typecheck (tsc --noEmit)
   - Unit tests (Vitest)
   - Build mobile (EAS preview build, optional, triggered by PR label)

2. **On merge to main:**
   - Web deploys automatically via Vercel
   - Mobile JS-only changes ship via EAS Update
   - Mobile native changes trigger EAS Build → EAS Submit to TestFlight + Play Internal Testing

3. **Manual workflows:**
   - Database migration verification (against a temp branch)
   - Edge function deployment (when functions change)

## Documentation discipline

`docs/` is part of the repository, version-controlled. Updates to docs go through PR review same as code.

- **Workdiary** (`docs/workdiary.md`) is the living log of every working
  session: tool inventory, decisions log, per-session entries. Read latest
  entry at session start; append entry at session end. Format template at
  the bottom of the file.
- **ADRs** (Architecture Decision Records) for any decision worth remembering: why we chose Expo, why we kept Supabase, why we deferred Peaq, etc. Format: short, numbered, dated, immutable (write a new ADR to supersede an old one).
- **API contracts** for every Edge Function: input shape, output shape, error cases, auth requirements.
- **Schema doc** is the human-readable companion to migrations. Updated in the same PR as a migration.
- **AI Agent Contract** is the single source of truth for the interface to the AI agent project.

## Local dev setup checklist

A new contributor (or Muhammed on a new machine) should be able to follow this and be running in under 30 minutes:

1. Install Node 22 (current Active LTS as of 2026; Node 20 went EOL 2026-04), pnpm 9
2. Clone repo
3. `pnpm install` at root
4. Install Supabase CLI
5. `supabase link --project-ref <dev-project-ref>`
6. Copy `.env.example` to `.env.local` in each app, fill in dev keys (from 1Password)
7. Install VS Code (free, from code.visualstudio.com)
8. Install the Claude Code extension from the VS Code Marketplace (official one by Anthropic)
9. Sign in with your Anthropic account when prompted
10. (Mobile) Install Expo Go on phone, scan QR from `pnpm --filter mobile dev`
11. (Web) `pnpm --filter admin dev` opens at localhost:3000

The `README.md` at the repo root contains exactly these steps.

## CLAUDE.md — Claude Code's project brief

`CLAUDE.md` at the repo root is read automatically by Claude Code (VS Code
extension and CLI) at the start of every session. It is a short brief — one
screen — that tells Claude what this project is, what stack it uses, where the
detailed docs live, what conventions to follow, and what's out of scope.

Keep it focused:
- The file is included in every Claude Code request, so length costs tokens
- Detailed content lives in `docs/` — `CLAUDE.md` points there, doesn't duplicate
- When you find yourself repeatedly correcting Claude on the same thing, add a
  line to `CLAUDE.md` so the next session starts knowing

Subfolder `CLAUDE.md` files (e.g., `apps/mobile/CLAUDE.md`) are supported for
area-specific conventions. Start with just the root one; add subfolder ones
only if a real need emerges.

Commit it to git so every Claude Code session in this repo (current and future) gets the same instructions.
